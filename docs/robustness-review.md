# DoubleDone robustness + security review

*Generated 2026-06-26 by a 7-lens adversarial code audit (error handling, input validation, auth, secrets/PII, sync integrity, Stripe/money, client concurrency). Every finding was re-checked by a skeptic and re-rated. Severity-ranked; each item tagged clear-fix (safe to batch) or needs-judgment (sign-off). Posture is solid for a small solo app; the work is data-integrity and cost, not breach.*

---

## Read

DoubleDone's security posture is solid for a small solo app, and the parts that matter most for money and auth are genuinely well built. The Stripe webhook signature verification, the premium gate (fails closed everywhere, pins algorithms, verifies against Supabase JWKS), the comp allowlist (read only after signature verification), and the MCP/RLS write path all hold up under scrutiny. Most of what the verifiers dropped were false alarms on those exact surfaces, which is a good sign. The real work sits in two places. First, data integrity in sync: the whole-row last-write-wins merge can silently erase a recurring-task completion or a slices progress count when another device makes a newer unrelated edit, and that is precisely the never-lose-a-task, never-shame-the-backlog failure the app exists to prevent. Second, cost protection on the AI backend: the free, no-auth AI routes accept unbounded input that forwards straight to Claude against a small shared budget, and the Stripe webhook has no idempotency or ordering guard, so a retry or out-of-order delivery can flip a real subscriber's entitlement. None of this is on fire. There is no data-exposure hole, no auth bypass, no forgeable premium. Fix the sync-merge data loss and the input caps first, add webhook idempotency next, then sweep the cheap clear-fix batch.

## Critical (fix before anything else)

Nothing rises to critical. There is no live data-exposure, auth-bypass, or forgeable-premium path. The highest-severity items are High and are about silent data loss and cost, not breach.

## High

- **Offline recurring completion and slices progress erased by a newer unrelated edit from another device.** `client/src/lib/sync-merge.ts:30-44` (whole-row LWW), `client/src/lib/sync.ts:44,69`, `client/src/lib/today.ts:36-38`. `completed_dates` and `slices` are synced fields, but the merge is whole-row last-write-wins, not field-level or set-union. Device A ticks a daily recurring task offline; device B later edits the same task's title (newer `updatedAt`) without that tick; on sync B's row wins and its `completedDates` overwrites A's, so the completion the user genuinely did disappears. `slices.done` truncates the same way. This is the exact shame-by-disappearance failure the app exists to prevent, and it is invisible: no error, no log. Fix: special-case these two fields in `mergeTasks` so completion is monotonic, union the two `completedDates` arrays (grow-only set) regardless of which row wins LWW, take `Math.max` of `slices.done`, attach to the winning row. Add a `sync-merge.test.ts` case proving an offline tick survives a remote-newer title edit, and record the decision in `decision-log.md`. (needs-judgment)

- **AI routes accept unbounded arrays and strings forwarded to Claude with no input-size cap.** `server/src/index.ts:736,483-484,582-589,867-869,329,374,639,780,822,685-694`; builders join the whole lot at `triage.ts:59`, `strategise.ts:61`, `combine.ts:47`, `lookbackSummary.ts:33`. `max_tokens` caps output but nothing caps input, where the cost is. The per-IP rate limit (`wrangler.jsonc:22`, 30 req/60s) caps request count, not size, so one accepted request can carry a multi-megabyte body that becomes a large input-token bill against the shared ~$25/mo Anthropic cap. The cheapest attack path is the free, no-auth routes gated only by an Origin check (`index.ts:258`) that a non-browser client bypasses by omitting the Origin header. `/ocr` already has a 1.9MB backstop and `/scrapbook/purge` a `.slice(0,200)` count cap, so the omission elsewhere is an oversight not a decision. Fix: a small shared helper, `capStrings(arr, maxItems, maxLen)` to slice array length (~100) and per-string length (~500), and `capText(s, maxLen)` for single-string routes (reject over ~2000 chars), applied uniformly in each route's parse block before the upstream fetch. Fold the MCP `add_task` title cap (`mcp.ts:180`) into the same helper so the authed path matches. These limits are unreachable in normal use. (needs-judgment)

- **Stripe webhook has no idempotency or ordering guard.** `server/src/stripe.ts:280-302,184-200`; `server/d1/schema.sql:46-55` (no processed-event table). `handleWebhook` verifies the signature then unconditionally calls `writeEntitlement` on every accepted event, with no event-id dedup and no event-time ordering guard. Stripe retries any non-2xx (and the handler returns 500 on a store error, so retries are guaranteed) and does not guarantee delivery order. A re-delivered older `customer.subscription.updated` "active" event can flip `cancel_at_period_end` back to false (silently un-cancelling a user who scheduled a cancel), or a late "deleted" event can revoke premium from a still-paying user, with no self-correction until the next genuine event. For a live app with real subscribers this is a trust and support problem. Fix: add a `processed_events(event_id text primary key, created_at)` table in D1, `INSERT OR IGNORE` the event id at the top of `handleWebhook` and return 200 early if already present; guard ordering by persisting the event timestamp on the entitlement row and only applying events at or after the stored one, or for `customer.subscription.*` re-fetch the subscription from Stripe by id and write its current state. Record the event id only after the write succeeds. (needs-judgment)

## Medium

- **`checkout.session.completed` grants premium on `status='complete'` even without captured payment.** `server/src/stripe.ts:153`. `const paid = obj.payment_status === 'paid' || obj.status === 'complete'`. A Checkout Session can reach `status='complete'` while `payment_status` is `unpaid` or `no_payment_required` (async/delayed payments, or a 100%-off promo). `allow_promotion_codes` is on (`stripe.ts:42`), so a fully-discounted code is a live path to premium without captured payment, spending the Anthropic budget on OCR/Sonnet routes. Fix: tighten to `const paid = obj.payment_status === 'paid'`, treat `checkout.session.completed` as attribution capture, and let the subsequent `customer.subscription.*` events carry ongoing active/trialing truth. Route intentional 100%-off comps through the comp allowlist, not the webhook. Confirm the trial flow still grants via the subscription `'trialing'` status before removing the OR. (needs-judgment)

- **Owner's personal email hardcoded in the comp allowlist in a PUBLIC repo.** `server/src/comp.ts:7`. `COMP_EMAILS = new Set(['[redacted]'])`, a real personal Gmail committed to a confirmed-public repo, now tied by name to a live commercial product. The premium-bypass angle is low (it needs a separate Gmail compromise plus a valid Supabase session, and only yields free Premium), so the genuine harm is PII disclosure: a personal email publicly linked to a named product, inviting spam and phishing. Fix: move the allowlist out of committed source, read `COMP_EMAILS` from a Worker secret (comma-separated, lowercased) or the D1-backed comp list the comments already anticipate. Immediate cheap mitigation: switch the owner's comp identity to a non-personal alias like `owner@doubledone.app`. Git history still holds the old value, but stopping further exposure and de-personalising is the win. (needs-judgment)

- **Remote-wins merge silently drops the local-only fields `big` and `manualOrder`.** `client/src/lib/sync-merge.ts:36`, `client/src/lib/sync.ts:59-78`, `client/src/lib/tasks.ts:37-38`. These two fields are deliberately not synced, so when the remote copy wins the merge (`merged.push(r)`), the merged task has no `big` and no `manualOrder` and is then persisted, losing both. The field comments claim they "survive sync (local wins)", which only holds when the local copy wins. Losing `big` makes a heavy day read as light (removing the relief offer and shrinking the celebration); losing `manualOrder` reshuffles a deliberately-ordered day. Both are premium single-device fields, so narrower blast radius than the recurring-completion loss, hence medium. Fix: in the remote-wins branch carry the local-only fields forward as an explicit allowlist, `{ ...r, ...(l.manualOrder != null ? { manualOrder: l.manualOrder } : {}), ...(l.big ? { big: l.big } : {}) }`, kept in one place so a future local-only field forces a decision. Add a test asserting both survive a remote-newer win. (needs-judgment)

- **First-sign-in sync is non-atomic: a partial push failure drops the pull-side merge and leaves the synced-owner marker stale.** `client/src/lib/sync.ts:88-106`, `client/src/app/index.tsx:250-255`. `syncOnce` pulls, merges, then does a single `pushTasks` upsert and only returns `merged` on full success; the orchestrator persists and calls `saveSyncedOwner` only after that resolves. If `pushTasks` throws partway (network drop, RLS rejection, payload too large), the whole pass rejects: the locally-resolvable merge including remote-newer pulls is dropped, and `saveSyncedOwner` never runs so the owner marker lags. On a large first-sign-in migration the upsert is one unbounded request, so some rows can land server-side while the client believes nothing synced. Mostly self-healing on the next clean sync, hence medium not high. Fix: compute and `saveTasks(merged)`/`setTasks(merged)` before the push so a push failure never loses the pull-side reconciliation; chunk `pushTasks` into batches (~100 rows); advance `saveSyncedOwner` only on full push success; add a `sync.push_partial` telemetry event. (needs-judgment)

- **`chart.tsx addTasks`: async load-modify-save with no busy guard.** `client/src/app/chart.tsx:96-114`, button at `179-185`. Unlike `suggest()` in the same file, `addTasks` has no in-flight flag and the button is disabled only on `selectedCount===0`. The `await loadTasks()` yields the loop, so a genuine double-tap starts two `addTasks`, both read the same on-disk store, both mint different-id sets, both `saveTasks` (full overwrite, last writer wins), dropping one minted set, and `router.replace('/')` fires twice. For a goal-charting flow the user explicitly accepted, silently losing the minted tasks is a trust break. Fix: add a busy flag mirroring `suggest()`, `if (adding || selected.length === 0) return; setAdding(true);`, and pass `disabled={adding || selectedCount === 0}`. (clear-fix)

- **Scrapbook "Make a scrapbook" button guarded only by React state, missing `disabled`.** `client/src/app/lookback.tsx:93-135`, button at `311-317`. `makeWeekScrapbook` guards on `bookBusy` (React state) and the button does not pass `disabled`, so two synchronous taps in the same frame both read `bookBusy===false` before the re-render lands, both call `makeScrapbook()`. Per the project's own gotcha, each Workers-AI image generation is roughly the entire daily free-tier neuron budget, so a same-frame double-fire is real billable waste plus a race between two `saveScrapbooks` writes for the same week. Fix: pass `disabled={bookBusy}` to close the visible window, and convert the guard to a ref (`bookBusyRef`) checked and set synchronously at the top of `makeWeekScrapbook` so a same-frame second call is rejected before any re-render. Do both. (clear-fix)

## Low / noted

- **NaN timestamp from a corrupt remote row wins every LWW comparison and permanently pins the task to the bad row.** `client/src/lib/sync.ts:64-65,70,75-76` (`Date.parse`, no NaN guard), `client/src/lib/sync-merge.ts:32`. `rowToTask` parses timestamps with no NaN guard, and the remote ingestion path bypasses the `tasks.ts` `withDefaults` backfill that protects local blobs. A NaN `updatedAt` makes `l.updatedAt > NaN` false, so the merge adopts the corrupt remote row over the good local copy, and every future comparison against NaN stays false, pinning the task to the bad row permanently. Also poisons the `byCreated` sort comparator (non-finite subtraction). Reachability is genuinely low today (the live column is `timestamptz not null`, the app stamps ISO itself), so it is a latent defensive gap, but the blast radius is silent task loss. Fix: guard each parse in `rowToTask` with `Number.isFinite`, falling back to `created_at` then `Date.now()`; harden `mergeTasks` to treat a non-finite `updatedAt` as the loser (`-Infinity`). Add a `rowToTask`/sync-merge test feeding an unparseable timestamp. (clear-fix)

- **`authHeader()` has no try/catch and is awaited outside the seams' own try blocks.** `client/src/lib/supabase.ts:31-36`; callers `ai.ts:226,248,286,317` and `stripe.ts:14,28`. `authHeader()` calls `await supabase.auth.getSession()` (reads AsyncStorage) with no try/catch, and in the premium seams the call sits outside the seam's try block. A rejection (storage corruption, native bridge error) escapes as an unhandled promise rejection instead of the intended calm empty/`[]`/`{ok:false}` fallback, potentially leaving the calling screen's busy state stuck. Affects every authed AI seam plus both Stripe seams identically. Fix: wrap `authHeader`'s body in try/catch returning null on any error; every caller's existing `if (!auth)` fallback then handles it cleanly. (clear-fix)

- **`handleEntitlement` does not wrap `readEntitlement`, returning an ungraceful 500 instead of a calm "free".** `server/src/stripe.ts:324` vs `premium.ts:82-87`. Dispatched as a bare return with no outer try/catch around the fetch body, so a transient D1 throw rejects the handler and the Worker returns a generic 500. This is the cosmetic client flag, not the money gate (which stays fail-closed), so no security impact, but the Premium/Settings screen can hard-500 on a D1 hiccup, brushing the never-alarm spine. Fix: wrap the `readEntitlement` call in try/catch and return the FREE entitlement shape (`premium:false`) on throw, mirroring `requirePremium`. (clear-fix)

- **`chart.tsx suggest()` relies on `chart()` never throwing, with no catch of its own.** `client/src/app/chart.tsx:75-88`. `suggest()` wraps the call in try/finally with no catch, correct today only because `lib/ai.ts chart()` swallows every error and returns `{ heading:'', steps:[] }`. That never-throws guarantee is load-bearing and implicit, and inconsistent with sibling seams in the same file that deliberately throw. A future refactor to `chart()`'s error contract would surface an unhandled rejection with no test catching it. Fix: add an explicit catch to `suggest()` setting the same calm line already shown for an empty course, decoupling the screen from the seam's internal contract. (clear-fix)

- **MCP tool errors echo the raw upstream Supabase HTTP status to the caller.** `server/src/mcp.ts:185,191,208`. `runTool` interpolates the raw status into the tool result (`Could not add it (${res.status}).`). It is only the numeric status, never a body or header, and the parallel REST surface collapses everything to a generic "upstream error". Minor backend-topology leakage, no secret or token exposed, genuinely trivial. Fix (optional consistency): map the status to a plain user-facing line without the raw code, the way `api.ts` does. (clear-fix)

- **`makeScrapbook` caption can carry an arbitrarily long model string with no length clamp.** `client/src/lib/ai.ts:364-374`, `server/src/scrapbook.ts:40-44` (`parseScene`); persisted at `storage.ts:212`, rendered at `lookback.tsx:297`. `parseScene` trims and de-quotes but applies no maximum length. Confirmed NOT a security issue: RN `<Text>` does not interpret HTML, so even a prompt-injected scene has no injection vector. The only residual is an oversized caption bloating the stored blob and distorting the polaroid layout. Fix: clamp in `parseScene` after trimming, `return scene.length >= 4 ? scene.slice(0, 200) : FALLBACK_SCENE`. (clear-fix)

- **No request-body byte ceiling before `request.json()` on any route.** `server/src/index.ts` (all handlers), `api.ts:214,247`, `mcp.ts:236`. Every handler parses the body with no Content-Length guard, so a large JSON body is materialised in Worker memory before any field-level validation. Bounded by the Cloudflare platform cap and the 30/min rate limit, and it shares the root cause of the input-cap finding (the field-level caps are the real backstop), hence low. Fix: a shared early guard at the top of `fetch()`, return 413 if Content-Length parses over ~2MB (the existing ~1.9MB budget for `/ocr`). Treat as defence-in-depth since Content-Length can be absent or spoofed. (clear-fix)

- **"Reflect on this week" has a state-only guard and `PremiumButton` ignores in-flight state.** `client/src/app/lookback.tsx:140-157`, button at `377-382`; `PremiumButton.tsx:7-33`. `reflectOnWeek` guards on `summaryBusy` (React state), the button does not pass `disabled`, and `PremiumButton` has no loading concept, so a same-frame double-tap fires two `lookbackSummary()` premium calls. Text is cheap so this is low, but it is a real billable double-call with no UI guard. Fix: pass `disabled={summaryBusy}` and keep the state guard, or convert to a `summaryBusyRef`. Higher-leverage: honour `disabled` at every `PremiumButton` call site (Plan my day / Chart / Reflect). (clear-fix)

- **`biteElephant` (via `breakdownExisting`) has no re-entry guard.** `client/src/app/index.tsx:870-885,981-984`; TaskRow Pressables at `TaskRow.tsx:150-153,218-228`. `biteElephant()` does not check `bdBusy` on entry; its BrainDump caller is guarded but `breakdownExisting()` (TaskRow "Break down" and the suggest-row prompt) calls it directly, and those Pressables have no disabled state. A double-tap fires two `clarify()` AI calls and two `breakdown.started` telemetry events. State is idempotent so the only harm is the wasted call and noisy telemetry. Fix: add `if (bdBusy) return;` as the first line of `biteElephant`, mirroring `bdSubmitQuestions`. (clear-fix)

- **MCP and REST write paths build `user_id` from an unverified JWT decode, relying solely on one Supabase RLS WITH CHECK.** `server/src/mcp.ts:79-91,175-183`, `api.ts:210-222`, `supabase/schema.sql:41-42`. On a task write the Worker decodes the bearer JWT without verifying its signature and puts the decoded `sub` straight into the row's `user_id`. Not exploitable today: with RLS enabled, the insert WITH CHECK (`auth.uid() = user_id`) rejects a forged id because `auth.uid()` comes from the token Supabase itself verifies downstream. The risk is purely latent: a future change disabling RLS or a column the WITH CHECK does not cover would silently turn the unverified decode into a cross-user-write primitive, because the Worker layer adds nothing. Fix (optional hardening): add a CI/ops probe asserting RLS is enabled and the four task policies exist, and have the two write paths verify the token signature (reuse `defaultVerifySub` from `premium.ts`) and use the verified sub. (needs-judgment)

- **`chart.tsx addTasks` navigates home after a swallowed save failure, silently losing the accepted plan.** `client/src/app/chart.tsx:96-114`. `saveTasks` (`storage.ts:43-49`) swallows AsyncStorage failures (logs, returns void), so `addTasks` cannot tell whether the write succeeded and navigates to Today unconditionally. On a storage failure (quota, disk error) the user charts a course, taps Add, lands on an empty Today, and the steps are gone with no signal, on a premium feature. Narrow (AsyncStorage failing on a handful of small objects is rare) but the navigation makes failure look like success. Fix: give `saveTasks` a success signal for this call site (return a boolean) and gate `router.replace('/')` on it; on failure keep the steps on screen and show a calm inline line. (needs-judgment)

- **`handleCheckout` attributes the Stripe session from an unverified JWT decode.** `server/src/stripe.ts:247` vs `premium.ts:44-59`. `handleCheckout` uses decode-only `decodeJwtSub`, unlike every other money write. An attacker can craft a token with an arbitrary `sub` and produce a real Checkout session attributed to that id, but they must pay real money to grant premium to an id they chose, which only helps if they already control that account. A correctness/consistency wart, the one unverified write in the Stripe chain, not an exploit. Fix: verify the token in `handleCheckout` with `defaultVerifySub`; confirm it does not regress the signed-out 401 behaviour. (needs-judgment)

- **No defensive handling of payment failure; the grace window depends on Stripe dunning config, not code.** `server/src/stripe.ts:157-172`. The code is correct (`premium = status === 'active' || status === 'trialing'`, so `past_due`/`unpaid` yields `premium=false` when a `customer.subscription.updated` fires). The residual is that during Stripe's dunning retries the subscription can stay `active` while the card is failing, with no `invoice.payment_failed`-driven early revoke, so a failing-renewal subscriber keeps consuming paid compute through the dunning window. Mostly an ops verification, not a code change. Fix: confirm in the Stripe dashboard that the endpoint is subscribed to `customer.subscription.updated` and `.deleted`; optionally handle `invoice.payment_failed` as an early defensive revoke if you want a tighter window. (needs-judgment)

- **Same-millisecond edits resolve as a tie that always adopts remote, dropping the local edit.** `client/src/app/index.tsx:78-80`, `sync-merge.ts:35-36`, test at `sync-merge.test.ts:46-50`. `updatedAt` has only millisecond resolution and the tie rule is "remote wins, push nothing", so two genuinely different edits to the same task on the same millisecond (rapid same-device, or skewed-clock cross-device) collide and the local edit is discarded with no content check. Rare for a near-solo-user app, but real silent loss, and the tie rule actively prefers losing the local change. Fix: make `updatedAt` strictly monotonic per device (`max(Date.now(), lastUpdatedAt + 1)`), or on a tie compare content and prefer-and-push local when any synced field differs. (needs-judgment)

- **Second anonymous device migrating into an existing account creates duplicate tasks.** `client/src/app/index.tsx:71-74`, `sync-merge.ts:38-40`. Task ids are device-local (not content-addressed), and the first-sign-in migration pushes every local-only task as a new row. A user who works anonymously on device A, signs in, then later re-types the same tasks anonymously on device B before signing into the same account gets duplicate rows because the ids do not collide. Not data loss, and rare, but undocumented, and clutters the calm Today screen. Fix: accept and document as a known limitation in `decision-log.md`, or de-dup at migration time by (title + due) within a window. (needs-judgment)

- **`sortDump` ("Sort for me") has no defence-in-depth guard at the state owner.** `client/src/app/index.tsx:1036-1050`. `sortDump()` calls `triage()` then `commit([...tasks, ...added])` with no guard of its own, relying entirely on the BrainDump child's `disabled`. The sole live caller is guarded, so there is no current bug; the note is that the non-idempotent append depends on a guard owned by the child rather than the state owner, so a future second caller or a child-guard regression would append twice. Fix (optional): a small in-flight ref in `index.tsx` so `sortDump` early-returns if a sort is already running. (needs-judgment)

## Clear-fix batch

These are the (clear-fix) items an orchestrator can safely apply now, in priority order:

1. **NaN-timestamp guard in `rowToTask` + `mergeTasks`** (`client/src/lib/sync.ts:64-65`, `sync-merge.ts:32`). Parse defensively with `Number.isFinite`, treat non-finite `updatedAt` as the loser, add a test. Protects against silent task loss.
2. **`chart.tsx addTasks` busy guard** (`client/src/app/chart.tsx:96-114`). Add an `adding` flag and `disabled` prop. Prevents dropping a minted task set on double-tap.
3. **Scrapbook button ref guard + `disabled`** (`client/src/app/lookback.tsx:93-135,311-317`). Closes a real per-image billable double-fire.
4. **`authHeader()` try/catch returning null** (`client/src/lib/supabase.ts:31-36`). Hardens every authed AI seam and both Stripe seams against an AsyncStorage rejection.
5. **`handleEntitlement` try/catch returning FREE** (`server/src/stripe.ts:324`). Keeps the Settings screen calm on a D1 hiccup.
6. **`chart.tsx suggest()` explicit catch** (`client/src/app/chart.tsx:75-88`). Decouples the screen from `chart()`'s implicit never-throws contract.
7. **"Reflect on this week" `disabled={summaryBusy}`** (`client/src/app/lookback.tsx:377-382`). Closes a premium double-call; ideally honour `disabled` at every `PremiumButton` call site.
8. **`biteElephant` re-entry guard** (`client/src/app/index.tsx:870`). Add `if (bdBusy) return;` to stop a duplicate `clarify()` call and telemetry event.
9. **`makeScrapbook` caption length clamp** (`server/src/scrapbook.ts:40-44`). Bound the stored blob and polaroid layout, `scene.slice(0, 200)`.
10. **MCP raw-status error message cleanup** (`server/src/mcp.ts:185,191,208`). Map status to a plain line without the raw code, matching `api.ts`.
11. **Request-body byte ceiling at `fetch()` entry** (`server/src/index.ts`). Shared 413 guard over ~2MB Content-Length, defence-in-depth behind the field caps.

## Verified-and-dropped

The verifiers filtered out several findings as non-issues, which is worth recording so the list reads as trustworthy rather than exhaustive:

- **The money and auth core held up.** `requirePremium` fails closed on every error path and pins algorithms to ES256/RS256 (blocking `alg:none` and HS256 key-confusion). The comp allowlist is read only after signature verification, so it cannot be spoofed. The client `devPremium` override is local-UI-only and inert in production. The Stripe webhook verifies HMAC with a timing-safe compare and a 300s tolerance, and is the only entitlement writer. The MCP/RLS write path is correct as shipped (the residual unverified-decode concern is filed as a latent low, not an exploit).
- **The MCP injection framing was a false alarm.** `complete_task`'s id goes through `URLSearchParams` (percent-encoded, no PostgREST filter injection), `add_task`'s title goes into a parameterised JSON body, and RLS is the real enforcement on `user_id`, so a tampered `sub` cannot escalate.
- **The feedback-email header-injection concern was not real.** The body is base64-encoded so CRLF cannot break into headers, all header fields are server constants, and Content-Type is pinned to `text/plain`.
- **Four client-concurrency "duplicate task" findings were false positives.** Combine-accept, the goodnight off-list note, and "I also did that" all run through a full-replace `commit()` reading a shared render closure, so a same-frame second tap overwrites rather than stacks, and the re-rendered path is guarded by a cleared-input check. Exactly one task survives in every case.
- **The account-gone R2 purge "401 race" was a false positive.** The purge route requires no auth, so `signOut()` cannot 401 it, and the image keys are captured before the local wipe, so no orphaning occurs on the realistic path.
- **Telemetry retaining raw task text is disclosed by design, not a bug.** It is pseudonymous (no user_id/IP/email), documented in the privacy policy and code headers, and intended for the moat. A retention cap is a reasonable product hardening to raise separately, not a code defect.
